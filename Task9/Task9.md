### **Название задачи:**
ADR-009: Настройка чтения с реплик и консистентность данных
### **Автор:**
Архитектурная команда "Мобильный мир"
### **Дата:**
21.03.2026
#### Контекст и описание проблемы
После внедрения шардирования и репликации MongoDB для онлайн-магазина "Мобильный мир" необходимо оптимизировать стратегию чтения данных. Каждый шард теперь представляет собой replica set с primary узлом для записи и secondary узлами для чтения. Требуется определить, какие операции могут безопасно читать данные с secondary реплик без риска для бизнес-логики, обеспечивая баланс между производительностью и консистентностью.

### **Обоснование выбора для каждой коллекции**

#### **1. Коллекция products**

**Операции, допускающие чтение с secondary:**
- *Просмотр каталога и поиск:* эти операции составляют ~70% всех запросов. Небольшая задержка (3-5 секунд) допустима, так как:
  - Изменения цен и описаний происходят относительно редко (несколько раз в день)
  - Пользователи не заметят задержку в несколько секунд при просмотре каталога
  - Значительно снижается нагрузка на primary узлы

**Операции, требующие primary:**
- *Проверка остатков при покупке:* критично для предотвращения продажи отсутствующего товара. Риск overselling может привести к отмене заказов и недовольству клиентов. Остатки обновляются часто (при каждой покупке).

#### **2. Коллекция orders**

**Операции, допускающие чтение с secondary:**
- *История заказов:* задержка до 10 секунд допустима, так как:
  - Пользователи редко просматривают историю сразу после оформления заказа
  - Это не влияет на бизнес-процессы
  - Позволяет разгрузить primary для обработки новых заказов

**Операции, требующие primary:**
- *Текущий статус заказа:* пользователи ожидают видеть актуальный статус. Особенно критично при статусах «оплачен», «отменён». Влияет на доверие к сервису и процесс выполнения заказа.

#### **3. Коллекция carts**

**Все операции требуют primary**, так как:
- Корзина — это активная рабочая область пользователя
- Любая задержка может привести к путанице (добавил товар, но не вижу)
- Частота обновлений очень высокая
- Объём данных относительно небольшой, что делает балансировку нагрузки менее критичной

### Конфигурация Read Preference в MongoDB

```shell
// Для операций с secondary чтением
db.products.find().readPref("secondaryPreferred", {
  maxStalenessSeconds: 5
})

// Для критичных операций
db.products.find({_id: productId}).readPref("primary")

// Настройка на уровне коллекции для orders
db.orders.find({
  customer_id: userId,
  created_at: {$lt: new Date(Date.now() - 60000)} // заказы старше минуты
}).readPref("secondary", {
  maxStalenessSeconds: 10
})

// Корзины всегда с primary
db.carts.find().readPref("primary")
```

### Мониторинг консистентности

```shell
// Метрика отставания репликации
db.adminCommand({
  replSetGetStatus: 1
}).members.forEach(member => {
  if(member.stateStr === "SECONDARY") {
    const lag = member.optimeDate - member.lastHeartbeat;
    if(lag > 5000) { // более 5 секунд
      alert(`High replication lag: ${lag}ms on ${member.name}`);
    }
  }
})
```

#### Преимущества выбранного решения

* Снижение нагрузки на primary узлы до 60%
* Повышение общей производительности системы
* Сохранение критически важной консистентности данных
* Масштабируемость чтения через добавление secondary узлов

### **Альтернативы**

| Альтернатива | Описание | Причина отказа |
|--------------|----------|----------------|
| **Все операции на primary** | Все операции чтения и записи только на primary узлы | Не решает проблему масштабирования, primary остаётся узким местом |
| **Все некритичные операции на secondary** | Максимальное использование secondary для всех операций чтения | Риск показа сильно устаревших данных для операций, требующих свежести |
| **Использование отдельного кеша (Redis)** | Внедрение Redis для часто запрашиваемых данных с асинхронным обновлением | Дополнительная сложность и затраты на инфраструктуру |
| **Разделение на отдельные БД для чтения и записи** | Создание выделенных кластеров master и slave с односторонней репликацией | Удвоение инфраструктурных затрат, сложность поддержания консистентности |
| **Использование Change Streams** | Асинхронное обновление материализованных представлений для чтения | Существенная задержка распространения, дополнительное хранилище |

### **Недостатки, ограничения, риски**

| Риск/Ограничение | Описание | Способ mitigation |
|------------------|----------|-------------------|
| **Усложнение логики приложения** | Разработчики должны явно указывать Read Preference для разных операций | Создание абстракций в коде, использование конфигураций на уровне подключения |
| **Отставание репликации** | При высоком replication lag secondary узлы могут возвращать устаревшие данные | Мониторинг lag с оповещениями, установка maxStalenessSeconds |
| **Риск показа устаревших данных** | При сбоях репликации или сетевых проблемах secondary может сильно отставать | Автоматический fallback на primary при превышении порогов задержки |
| **Требуется обучение команды** | Разработчики должны понимать последствия выбора разных уровней консистентности | Документация best practices, код-ревью с проверкой использования Read Preference |
| **Сложность отладки** | Разные узлы могут возвращать разные данные, что усложняет воспроизведение проблем | Внедрение трассировки запросов с указанием узла, на котором выполнялся запрос |
